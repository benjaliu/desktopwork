## 一、产品概述

### 1.1 产品定位

DesktopWork 是一款**本地运行的桌面工作台平台**，类似飞书工作台在桌端的实现。平台本身是一个容器，提供 AI 对话、工具集成、本地存储等基础能力；具体的「工作内容」以 App 的形式插入到平台中。

与飞书/钉钉等云端协作平台不同，DesktopWork 的所有数据、处理过程和配置都保存在本地，用户拥有完全的控制权和隐私保护。

### 1.2 核心价值

| 价值维度 | 描述 |
|---------|------|
| **隐私优先** | 所有数据在本地处理，不上传任何数据到第三方服务器 |
| **本地执行** | 不依赖云端服务，AI 推理在本地完成（或用户自控的 API） |
| **平台能力** | 工作台架构，支持集成多种 App，类似飞书工作台 |
| **AI 深度集成** | AI 不是单独的功能，而是渗透到平台和各 App 的基础能力 |
| **开源透明** | 核心代码开源，用户可审计、可定制 |

### 1.3 目标用户

- **开发者** — 需要 AI 辅助编程、终端操作、系统管理
- **知识工作者** — 需要 AI 协助文档处理、信息整理、任务自动化
- **隐私敏感用户** — 不希望数据经过第三方服务器
- **效率追求者** — 希望在同一桌面应用中完成多种工作，减少应用切换

### 1.4 与飞书的关系

DesktopWork 不是飞书的替代品，而是**飞书工作台模式的本地化实现**。

| 维度 | 飞书（云端） | DesktopWork（本地） |
|------|------------|-------------------|
| 数据存储 | 云端服务器 | 本地文件系统 |
| 部署方式 | SaaS | 桌面应用 |
| 网络依赖 | 必须在线 | 离线可用（核心功能） |
| 定制性 | 受限 | 完全开源可定制 |
| App 生态 | 飞书开放平台 | 本地 App 目录 |

### 2.2 App（应用）

App 是 DesktopWork 平台上的工作单元。每个 App 由三部分组成：

| 组成部分 | 描述 | 位置 |
|---------|------|------|
| **前端界面** | 用户看到和交互的 UI | React 组件 / HTML |
| **后端 API** | 业务逻辑和数据处理 | desktop-agent 路由 |
| **平台调用** | 使用平台基础能力 | Auth、Storage、Skills 等 |

### 2.3 App 实现方式（界面来源）

App 的前端界面有两种来源，平台统一通过认证和路由进行管理：

```
┌─────────────────────────────────────────────────────────┐
│                   平台层（统一认证 + 路由）                │
├────────────────────────┬────────────────────────────────┤
│   内置 App             │   外部集成 App                  │
│   （界面在平台内部实现） │   （界面由外部应用提供）         │
├────────────────────────┼────────────────────────────────┤
│                        │                                │
│  desktop-agent 直接    │  平台通过 iframe / WebView      │
│  渲染 React 组件        │  加载外部应用的 UI              │
│                        │                                │
│  例：Bot Chat          │  例：集成飞书日历 Web 版         │
│  例：Settings          │  例：集成 Notion 页面            │
│                        │                                │
│  UI 在同一进程内        │  UI 在独立进程中，通过 URL 加载  │
│  响应速度快             │  享受外部应用原生体验             │
│  平台完全控制           │  外部应用保持独立更新             │
└────────────────────────┴────────────────────────────────┘
```

| 维度 | 内置 App | 外部集成 App |
|------|---------|------------|
| 界面实现 | desktop-agent 内 React 组件 | 外部应用提供的 HTML/WebView |
| 后端 API | desktop-agent 路由 | 外部应用自己的 API |
| 更新方式 | 随 desktop-agent 更新 | 外部应用独立更新 |
| 数据存储 | 本地 JSONL / 文件 | 外部应用决定 |
| 认证 | 统一认证体系 ✅ | 统一认证体系 ✅（OAuth / SSO） |
| 路由 | 平台路由（`/app/:id/*`） | 平台路由 → iframe/WebView 加载外部 URL |

**说明：**

- **内置 App** 的界面代码随平台一起发布，与平台深度集成，适合核心功能和需要本地数据处理的能力。
- **外部集成 App** 通过标准 WebView 或 iframe 嵌入外部应用的界面，适合已有 Web 应用的集成，外部应用只需适配平台认证协议即可接入。
- 两种方式共用平台的认证体系，用户只需登录一次。

> ⚠️ **MVP 阶段说明**：MVP（v0.1）只实现内置 App。外部集成 App 的接入机制在 v0.5+ 再考虑。

**App 的生命周期：**

| 阶段 | 描述 |
|------|------|
| **注册** | App 在平台注册，提供 name、icon、routes、permissions |
| **激活** | App 被用户打开或设为默认 |
| **运行** | App 在工作台中展示，可交互 |
| **挂起** | App 被最小化或切换到后台（保持状态） |
| **卸载** | App 被移除（数据可选择保留） |

### 2.4 Skills（技能）

Skills 是 App 的能力扩展单元，类似于 VS Code 插件市场的插件。

- Skills 可以被多个 App 共享
- Skills 通过 MCP 协议暴露工具能力
- 平台提供系统级 Skills（文件操作、终端、通知）

### 2.5 Memory（记忆）

Memory 是平台级别的持久化存储，供所有 App 使用：

- **Session 记忆** — 单个对话的历史
- **知识库记忆** — 跨会话的长期记忆（向量 + 图谱）
- **App 状态** — 各 App 的持久化状态

### 2.6 App 分类（安装来源）

| 类型 | 说明 | 示例 |
|------|------|------|
| **平台内置 App** | 随平台发布的核心 App | Bot Chat、Settings |
| **本地 App** | 安装在本地的 App | 从 `~/.config/desktopwork/apps/` 加载 |
| **外部集成 App** | 外部应用界面嵌入 | 未来集成飞书 Web 版等（v0.5+） |

> ⚠️ **MVP 阶段说明**：MVP（v0.1）只实现平台内置 App。外部集成 App 在 v0.5+ 再考虑。

## 三、内置 App

### 3.1 Bot Chat（AI 对话）

**定位：** 平台内置的第一个 App，也是 MVP 的核心功能。

**功能：**

| 功能 | 描述 |
|------|------|
| 自然语言对话 | 与 AI Bot 进行自由对话 |
| 流式响应 | 实时流式输出，打字机效果 |
| 多会话管理 | 多个独立会话，互不干扰 |
| 上下文记忆 | 同一会话内记住对话历史 |
| 工具调用 | Bot 可以调用 Skills 执行实际操作 |
| 代码高亮 | 代码块自动语法高亮 |
| Markdown 渲染 | 支持完整 Markdown |

**与飞书机器人的关系：** Bot Chat 等同于飞书里的「机器人」功能，是 AI 能力的入口。

**用户交互流程：**

```
用户输入消息
    ↓
Bot Chat App
    ↓
AI 推理引擎 + Skills 调用
    ↓
流式返回结果
    ↓
存入 Session 历史 + 更新 Memory
```

## 四、平台基础能力

### 4.1 认证（Auth）

| 功能 | 描述 |
|------|------|
| 首次使用引导 | 首次启动时的 LLM 配置向导 |
| 配置管理 | 管理 AI Provider 配置（API Key、Model 等） |

**说明：** DesktopWork 是本地应用，认证主要是区分「配置管理员」和「访客」，不涉及多用户。

### 4.2 配置（Config）

统一的配置管理，所有 App 共享：

| 配置项 | 分类 | 描述 |
|--------|------|------|
| LLM Provider | 全局 | OpenAI / Anthropic / 本地模型 |
| LLM Model | 全局 | 模型名称 |
| LLM API Key | 全局 | API 密钥 |
| LLM Base URL | 全局 | API 端点（支持代理） |
| 数据目录 | 全局 | 配置和数据存储位置 |
| 开机自启 | 系统 | 是否开机自动启动 |
| 通知 | 系统 | 是否启用系统通知 |

### 4.3 存储（Storage）

| 数据类型 | 存储位置 | 描述 |
|---------|---------|------|
| Session 历史 | `~/.local/share/desktopwork/sessions/` | 对话消息记录 |
| 知识库 | `~/.local/share/desktopwork/memory/` | 长期记忆、向量化检索 |
| App 状态 | `~/.local/share/desktopwork/app-data/` | 各 App 独立存储 |
| Skills 配置 | `~/.config/desktopwork/skills/` | Skills 清单和脚本 |
| 用户配置 | `~/.config/desktopwork/config.json` | 全局配置 |
| 日志 | `~/.local/share/desktopwork/logs/` | 运行日志 |

### 4.4 Skills 系统

Skills 是平台暴露给 AI 的工具能力：

| 功能 | 描述 |
|------|------|
| Skill 加载 | 从本地 Skills 目录加载 |
| Skill 启用/禁用 | UI 可控制每个 Skill 的开关 |
| 协议 | Skills 通过标准协议暴露为工具 |
| 系统 Skills | 平台内置的文件操作、终端、通知等 |

### 4.5 IPC（进程间通信）

desktop-agent（Node.js）与 Tauri（Rust）之间的通信：

| 通道 | 方向 | 用途 |
|------|------|------|
| HTTP | Tauri → agent | UI 调用 App API |
| SSE | agent → Tauri | 实时推送（流式响应） |
| stdin/stdout | agent ↔ Tauri | 进程管理信号 |

## 五、运行架构（高层概念）

### 5.1 整体构成

DesktopWork 运行时由三个角色组成：

| 角色 | 职能 |
|------|------|
| **Tauri 运行时** | 窗口管理、托盘、原生能力（通知、文件选择等） |
| **Platform 进程** | App 容器、平台 API、Session 存储、Skills 加载、AI 引擎调用 |
| **LLM Provider** | 用户配置的 AI 模型服务（Anthropic / OpenAI / 任意兼容 provider） |

三个角色通过本地 HTTP / IPC 通信，平台进程作为业务中台，只与本地的 Tauri 运行时交互，不需要暴露到公网。

### 5.2 App 容器模型

平台提供统一的 App 容器：

- **内置 App** — 平台内置发布的 App，界面与平台同一进程渲染
- **外部集成 App** — 外部应用界面嵌入（v0.5+ 考虑），通过标准容器加载

所有 App 共享平台认证体系，用户一次登录，多个 App 可用。

### 5.3 App 注册机制

每个 App 需要在平台注册，描述自己的身份、图标、路由和所需权限，平台根据注册信息决定如何加载和展示该 App。

### 5.4 数据流（Bot Chat 场景）

```
[用户在 Bot Chat 输入消息]
    ↓
[前端 App] → 平台 API
    ↓
[Platform 进程] → AI 引擎调用
    ↓
[AI 引擎] → LLM Provider（用户配置的 API）
    ↓ 流式响应
[Platform 进程] → 前端渲染
    ↓
[Session 存储] + [Memory 更新]
```
